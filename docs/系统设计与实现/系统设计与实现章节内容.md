# 系统设计与实现

## 5.1 系统开发环境

本系统采用 B/S 架构进行设计与实现，用户通过浏览器访问系统页面，服务器端负责业务逻辑处理、数据访问和页面渲染。系统后端基于 Python 语言和 Flask Web 框架开发，使用 SQLAlchemy 作为 ORM 数据访问工具，数据库采用 MySQL 存储用户、图书、借阅、座位预约、图书漂流、收藏评价等业务数据。前端页面主要采用 HTML、CSS、JavaScript 和 Jinja2 模板技术实现，通过 Flask 模板引擎将后端数据动态渲染到页面中。

表 5-1 系统开发环境

在依赖管理方面，系统主要使用 Flask、Flask-SQLAlchemy、Werkzeug、python-dotenv 等第三方库。其中 Flask 用于处理 Web 请求和路由分发，Flask-SQLAlchemy 用于简化数据库模型定义和数据库操作，Werkzeug 用于密码哈希加密与校验，python-dotenv 用于辅助读取环境变量配置。

## 5.2 系统框架设计与配置

### 5.2.1 系统总体框架设计

本系统按照 MVC 思想进行分层设计，将页面展示、业务处理和数据持久化分离。用户在浏览器中发起请求后，Flask 路由接收请求并调用对应业务处理函数；业务处理函数根据请求参数和当前登录状态完成权限校验、数据查询、数据写入等操作；最终通过 Jinja2 模板将处理结果返回给浏览器。

系统主要由表现层、控制层、业务层、数据访问层和数据存储层组成。表现层负责页面展示和用户交互，包括登录页面、图书列表页面、图书详情页面、座位预约页面、图书漂流页面、个人中心页面等。控制层由 Flask 蓝图路由组成，负责接收请求、分发功能和返回响应。业务层负责实现登录认证、图书借阅、座位预约、收藏评价、图书漂流、智能检索和统计分析等核心逻辑。数据访问层通过 SQLAlchemy 模型完成数据库读写。数据存储层使用 MySQL 保存系统运行所需的业务数据。

### 5.2.2 Flask 蓝图模块设计

为了降低单个路由文件的复杂度，系统采用 Flask 蓝图机制对功能进行模块化划分。各业务模块分别放置在 `app/routes` 目录下，并在应用工厂函数中统一注册。系统主要蓝图模块如表 5-2 所示。

表 5-2 系统蓝图模块说明

| 蓝图模块 | URL 前缀 | 功能说明 |
|---|---|---|
| main | / | 系统首页、数据概览和入口跳转 |
| auth | /auth | 登录、注册、退出、修改密码、头像上传、用户管理和批量导入 |
| books | /books | 图书列表、图书详情、借阅归还、收藏评价、推荐和管理员图书维护 |
| seats | /seats | 座位列表、座位预约、座位释放和超时预约清理 |
| crossing | /crossing | 图书漂流发布、浏览、申请、处理和我的漂流 |
| ai | /ai | AI 智能检索、图书匹配和统计报告生成 |

蓝图划分使系统的功能边界更加清晰。例如，用户认证相关逻辑集中在 `auth.py` 中，图书借阅与评价相关逻辑集中在 `books.py` 中，座位预约逻辑集中在 `seats.py` 中，图书漂流逻辑集中在 `book_crossing.py` 中。这样的结构便于后续维护和功能扩展。

### 5.2.3 系统配置设计

系统配置集中保存在 `config.py` 文件中，主要包括 Flask 安全密钥、数据库连接地址、SQLAlchemy 配置和 AI 服务接口配置等内容。数据库连接采用 MySQL 连接字符串，格式为 `mysql+pymysql://用户名:密码@主机地址:端口/数据库名?charset=utf8mb4`，通过 `charset=utf8mb4` 保证中文数据能够正常存储和显示。

系统启动入口为 `run.py`。启动时，系统首先调用 `create_app()` 创建 Flask 应用实例，然后在应用上下文中执行数据库建表、字段兼容检查和基础数据初始化。系统通过 `db.create_all()` 根据模型自动创建数据表，并调用结构兼容方法补充已有数据库中缺失的字段。为了便于系统演示和测试，启动时还会插入默认账号、图书、座位、借阅记录、图书漂流、收藏评价等样例数据，并通过唯一字段判断数据是否已存在，避免重复插入。

## 5.3 前台用户功能模块实现

前台用户主要指学生和教师等读者用户。由于学生和教师在前台可使用的功能基本一致，因此系统在前台功能实现中将其抽象为统一用户角色。

### 5.3.1 用户登录与注册功能实现

登录功能是前台用户进入系统的入口。用户在登录页面输入账号和密码后，系统会依次按照学生学号、教师工号和普通用户名进行匹配。若账号存在并且密码校验通过，系统将账号编号、账号类型、用户名、角色和头像路径写入 Session，用于后续页面展示和权限控制。若账号或密码错误，系统会返回错误提示并要求用户重新输入。

注册功能面向学生和教师用户。学生注册时需要填写学号、姓名、性别、专业和密码；教师注册时需要填写工号、姓名、性别、专业或研究方向和密码。系统在保存前会检查学号或工号是否已经存在，避免重复注册。用户密码不以明文形式保存，而是通过 Werkzeug 的哈希函数加密后写入数据库，提高账号安全性。

图书检索功能用于帮助用户快速查找馆藏图书。用户进入图书列表页面后，可以查看系统中的全部图书，也可以根据书名、作者、ISBN 或图书分类进行筛选。系统根据用户输入的查询条件构造数据库查询语句，并将查询结果以列表形式展示在页面中。每条图书记录显示书名、作者、分类、馆藏位置、借阅状态和封面图片等信息。

### 5.3.3 图书借阅与归还功能实现

前台用户可以在图书列表页面或图书详情页面借阅状态为可借的图书。用户点击借阅按钮后，系统首先检查用户是否已登录，然后读取目标图书信息并判断图书状态是否为 `available`。若图书可借，系统创建借阅记录，并根据当前账号类型写入 `user_id`、`student_id` 或 `teacher_id`，同时将图书状态更新为 `borrowed`。若图书已经被借出，则系统提示该图书暂不可借。

用户可以在“我的借阅”页面查看个人借阅记录。归还图书时，系统根据借阅记录编号查询对应记录，并校验该记录是否属于当前用户。校验通过后，系统将借阅记录状态更新为 `returned`，记录归还时间，并将图书状态恢复为 `available`。这种设计保证了借阅记录和图书状态同步变化，避免出现图书状态与实际借阅情况不一致的问题。

### 5.3.4 座位预约功能实现

座位预约模块用于实现图书馆学习座位的在线查看和预约。用户进入座位预约页面后，可以按照楼层、区域和座位状态查看座位分布。系统在页面中显示座位号、所在区域、是否有电源和当前状态，使用户能够直观选择合适座位。

用户点击预约按钮后，系统会先检查当前用户是否已有有效预约。如果用户已经存在 `active` 状态的预约记录，则系统不允许再次预约，避免同一用户同时占用多个座位。若用户没有有效预约且目标座位为空闲，系统创建座位预约记录，并将座位状态更新为 `occupied`。用户离开座位时可以主动释放座位，系统会将预约记录状态改为 `completed`，并将座位状态恢复为 `free`。

系统还实现了超时预约自动释放功能。用户进入座位页面时，系统会检查所有处于 `active` 状态且预约时间超过规定时长的记录，自动结束这些预约并释放对应座位，提高座位资源利用率。

### 5.3.5 图书收藏与评价功能实现

图书收藏功能用于记录用户感兴趣的图书。用户在图书详情页面点击收藏按钮后，系统会在 `book_favorites` 表中保存收藏记录，并绑定当前用户账号和图书编号。用户可以在“我的收藏”页面查看所有已收藏图书，也可以取消收藏。

图书评价功能与借阅归还流程关联。系统要求用户必须归还某本图书后才可以对该图书进行评价，并且每个用户对同一本图书只能评价一次。用户提交评价时需要选择 1 至 5 星评分并填写文字内容。评价提交后默认进入 `pending` 状态，等待管理员使用 AI 审核。只有审核通过的评价才会展示在图书详情页面，避免不合规内容直接公开显示。

### 5.3.6 图书漂流功能实现

图书漂流模块用于实现校园闲置图书共享。用户可以在漂流页面发布自己的闲置图书，填写书名、关联课程、新旧程度和补充说明等信息。发布成功后，图书进入漂流列表，其他用户可以查看漂流图书详情。

用户在漂流详情页面可以提交领取申请，并填写给提供者的留言。系统会检查申请人是否为图书提供者本人，防止用户申请自己发布的图书；同时会检查是否已经对同一本漂流图书提交过申请，避免重复申请。图书提供者或管理员处理申请后，系统更新申请状态和漂流图书状态。当某个申请被同意后，图书状态变为已领取，同一图书的其他申请会被拒绝。

### 5.3.7 个性化推荐与 AI 智能检索功能实现

个性化推荐模块根据用户借阅记录和收藏记录分析用户偏好。系统统计用户经常借阅或收藏的图书分类，并排除用户已经借阅或收藏过的图书，生成个性化推荐列表。同时，系统还根据全馆借阅次数、收藏次数和图书入库时间生成热门借阅推荐、热门收藏推荐和最新图书推荐。这样既能体现用户个人阅读偏好，也能保证推荐内容的丰富性。

AI 智能检索模块用于支持自然语言检索。用户可以输入类似“推荐几本适合学习 Python 的书”这样的自然语言需求。系统会先读取本地馆藏图书数据，组织书名、作者、分类、状态等上下文信息，然后调用外部 AI 服务进行图书匹配。AI 返回结果后，系统将推荐内容展示在页面中。该功能要求 AI 只能基于本地馆藏进行推荐，避免生成系统中不存在的图书。

### 5.3.8 摘要与报告生成模块实现

摘要与报告生成模块用于对系统运行数据进行统计分析和文本化总结。系统在 `/ai/report` 页面中提供馆藏与空间大数据分析功能，该页面主要面向管理员开放。管理员进入数据大屏后，系统会统计近 30 天借阅趋势、图书分类借阅分布、座位预约热力数据、馆藏总数、累计借阅次数、当前借阅数量、座位占用率等信息，并通过 ECharts 图表进行可视化展示。

在摘要生成方面，系统根据数据库中的图书数量、借阅记录、座位状态和预约高峰等数据自动组织运营摘要。例如，系统会计算当前馆藏总数、累计借阅次数、热门借阅分类、座位占用率和高峰预约时段，并生成一段简短的管理建议。该摘要能够帮助管理员快速了解图书馆当前运行情况，为补充热门分类图书、优化座位管理和调整服务安排提供参考。

在报告生成方面，系统提供 `/ai/api/generate-report` 接口。用户或管理员可以选择图书作为报告对象，也可以输入自定义主题，并设置报告类型和篇幅要求。系统将目标书名、作者、报告类型、篇幅要求等信息组织为提示词，调用 DeepSeek API 生成 Markdown 格式的结构化报告。若用户对已有报告不满意，还可以提交优化要求，系统会在原报告基础上重新生成修改后的完整报告。该功能使系统不仅能够完成基础数据展示，还能够对馆藏资源和运营数据进行进一步的文本分析。

### 5.3.9 个人中心功能实现
